# Toto Wolff v2 — Governance Protocol as Infrastructure

## Problem Statement

The real problem is not "engineers need a better CLAUDE.md." The real problem is: **Claude Code sessions are ephemeral and non-collaborative**. Every engineer starts from zero. Council rulings disappear when the session ends. P10 plans are files no one reads. Personas live on one machine. There is no team memory.

The current solution (a shell script that wires CLAUDE.md) solves the individual installation problem. It does not solve institutional memory, collaboration, or automation.

## Architecture Decision

**Approach C: Hybrid strangler fig** — confirmed and extended by expansion ceremony.

> [!note] PREMISE CORRECTION 2026-06-15
> The seam is **not** a gstack dependency on council/p10. Council and p10 are **repo-native custom skills** in `~/.claude/skills/llm-council/` and `~/.claude/skills/p10/` — they have never been gstack skills and run without it. What the v2 MCP server actually replaces is the **model-routing + persistence** layer (the council/p10 implementations that today run as host-spawned subagents), making the protocol portable beyond a single Claude Code session. See `Council/Congressional-Records/2026-06-12-does-the-governance-server-need-a-brain.md` for the latest ruling (server is a gate, not a brain; intelligence stays host-native).

The seam is the host-native subagent routing of council/p10 logic. We replace exactly that with a TypeScript MCP server. Everything else (ship, review, browse) stays in gstack.

```
toto (TypeScript monorepo)
├── packages/
│   ├── core/          # Vault operations, council/p10 logic (~500 LOC)
│   ├── mcp-server/    # MCP server exposing governance tools (~300 LOC)
│   ├── cli/           # toto binary — setup, doctor, whoami, search, last, audit (~600 LOC)
│   ├── linear-sync/   # Linear API bidirectional sync (~400 LOC)
│   └── dashboard/     # React web dashboard (~800 LOC)
├── personas/          # Persona packages (npm registry)
└── setup              # Shell bootstrapper — kept for backward compat
```

**gstack stays for:** ship, review, browse, qa, cso, all commodity skills.
**gstack replaced for:** council, p10, vault-write, vault-search — these move to the MCP server.

## Vision

### 10x Check
The 10x version of Toto is a governance protocol ambient in every engineering workflow. You open a PR and within 60 seconds, @toto-wolff comments: "This touches the auth service boundary. Council ruling from 2025-11-14 says no auth changes without a p10 plan. Auto-generated p10 at [link] — review before merging." A new engineer runs `brew install toto && toto init` and has the full institutional memory of every engineering decision the team has made.

### Platonic Ideal
Toto feels like a silent team lead who has read every decision the company has ever made and quietly surfaces the relevant one at exactly the right moment. Not a chatbot. A protocol. Every decision is timestamped, searchable, auditable, and linked to the Linear ticket it created.

## Implementation Phases

### Phase 1 — Foundation (1-2 weeks)
Build `packages/core` and `packages/mcp-server`. This is the seam replacement:
- `vault_read(query)` — search vault by keyword/tag
- `vault_write(path, content, frontmatter)` — write a vault file atomically; performs `git add + git commit` atomically per write
- `council_run(question, context)` — spawn the tiered council (Haiku scouts, Sonnet analysts, Opus chairman)
- `p10_plan(task, phase, council_ref)` — run p10 analysis and draft plan

**Exit gate**: MCP server passes 2-week parallel run → remove `/council` and `/p10` from `CLAUDE.md` skill routing.

Ship: MCP server runs locally, registers with Claude Code, council/p10 work without gstack.

### Phase 2 — CLI (1 week)
Build `packages/cli`:
- `toto init` — reads `~/.claude/settings.json`, merges MCP server entry (read-then-merge, not overwrite); stores Linear + Anthropic keys in OS keychain via `keytar`
- `toto doctor` — validates keychain entries, MCP registration, vault git status, open conflicts
- `toto whoami` — active persona, vault path, pending p10s
- `toto search "<query>"` — vault search via ripgrep
- `toto last` — last 5 decisions
- `toto audit` — stale/orphaned/contradicted rulings

Ship: `npm install -g @toto-wolff/cli` or `brew install toto-wolff/tap/toto`.

### Phase 3 — Integrations (2 weeks)
**Pre-condition:** Linear integration spec (T8 in TODOS) must be written and approved before Phase 3 starts.

Build `packages/linear-sync`:
- `linear_sync(council_record)` — create/update Linear issue from council ruling
- `linear_link_p10(p10_path, linear_id)` — link p10 plan to existing ticket
- `toto status` — show open decisions with Linear status
- Error handling: deleted Linear issue → log + skip (don't fail council write); 429 → exponential backoff, max 3 retries

### Phase 4 — Dashboard + Personas (2 weeks)
Build `packages/dashboard` and persona registry:
- React dashboard served by MCP server at `localhost:PORT`
- `toto dashboard` — opens browser
- Persona packages: `packages/personas/engineering`, `packages/personas/devops`, etc.
- `toto persona add <your-org>/engineering` — install from npm registry
- All 4 dashboard views require intentional empty-state copy (T9 in TODOS)

## Review Decisions (resolved 2026-06-05)

1. **Scope**: All 5 accepted items in scope. Throughput assumption: multiple p10/karpathy agents during development.

2. **Migration cutover (Phase 1 exit gate)**: Remove `/council` and `/p10` from `CLAUDE.md` AFTER functional verification of BOTH services: (1) run `council_run` on 3 real decisions — Congressional Records must match gstack output (ruling, confidence, rationale, timestamp ISO 8601, council_tier[]); (2) run `p10_plan` on 3 real tasks — output must include status, stages[], vault_path, timestamp; (3) test P10BlockedError path with an adversarial prompt. On 2/3 fail for either service: open GitHub issue, do NOT cut over. Both services must pass independently.

3. **Vault write model**: Append-only log files. Each council session and p10 plan writes to its own dated file only. No shared mutable INDEX.md — replaced by `vault_search` (ripgrep). `git add + git commit` per file for audit trail. No shared mutable file = no git conflict possible.

4. **API key storage**: OS keychain via `keytar`. Keys never touch disk as plaintext.

5. **MCP server registration**: `toto init` uses read-then-merge on `settings.json`. Graceful fallback with copy-pasteable JSON if schema unrecognized.

6. **Test strategy**: Vitest unit (packages/core, packages/mcp-server) + bats E2E (packages/cli, ported from existing 42-test suite).

7. **Linear sync value prop**: Linear is the stakeholder visibility layer — leadership and PMs who don't run Claude Code sessions. Not for engineers (who have MCP access). Phase 3 scope scoped accordingly: council ruling → Linear issue for stakeholder tracking, not engineer workflow.

### Failure modes

| Failure | Detection | Mitigation |
|---|---|---|
| Git lock race on concurrent vault_write | `fatal: Unable to create .git/index.lock` | In-process write queue (D3) serializes all writes |
| ripgrep not installed | vault_read returns no results silently | Pre-flight check in vault_read → ConfigurationError (D10); CI workflow installs rg |
| keytar returns null (no key in keychain) | council_run starts then 401s mid-chain | Null-safe fallback: `(await keytar.getPassword(...)) ?? process.env.ANTHROPIC_API_KEY ?? throw ConfigurationError` (D15) — same fallback applies to p10_plan |
| Anthropic API key missing entirely | ConfigurationError thrown | Pre-flight check in council_run AND p10_plan before first API call (D5) |
| MCP server port 2727 in use | Server fails to bind silently | Port conflict → clear error message to stderr (D2) |
| settings.json schema unrecognized by toto init | Silent overwrite of existing config | Read-then-merge + graceful fallback with copy-pasteable JSON (D4) |
| Phase 1 exit gate fails (2/3 council sessions don't match gstack output) | Eval harness comparison | Open GitHub issue, do NOT cut over; both systems remain active (D14) |
| Phase 1 exit gate fails (p10_plan output malformed or missing required fields) | eval/p10-plan.eval.ts | Same rule: open GitHub issue, do NOT cut over. p10_plan success criteria: status field present, stages array non-empty, vault_path written, timestamp ISO 8601 |
| P10Service Opus returns BLOCKED status | p10_plan throws P10BlockedError | Do NOT vault_write a draft plan without Opus approval. Surface P10BlockedError to MCP caller with Opus rationale. Engineer must escalate to /council. |
| launchd plist not loaded after install | MCP server not running | install-daemon.sh runs `launchctl load` after writing plist (D11) |
| Git timeout on slow vault repo | vault_write hangs | 10s git operation timeout; surface as VaultWriteError |

### Parallelization strategy (Phase 1)

Phase 1 has two workstreams that can proceed in parallel after monorepo scaffold:

**Track A — VaultService (independent):** `vault_write`, `vault_read`, write queue, ripgrep integration. No Claude API dependency. Fully buildable and unit-testable before Track B.

**Track B — Council/P10 service:** `CouncilService`, `P10Service`, `keytar` integration. Blocked only on `AnthropicClient` interface contract (define first), not on Track A completion.

**MCP server (joins both):** Thin adapter wiring Track A + B service classes to MCP tool handlers. Unblocked once both tracks export their service interfaces.

Recommended: scaffold monorepo + define `AnthropicClient` interface in one session, then run Track A and Track B as independent CC sessions in parallel.
